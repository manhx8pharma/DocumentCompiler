Đề xuất kiến trúc Chorus Block
Phát hiện quan trọng từ nghiên cứu codebase
Trước khi vào thiết kế, có 3 điều quan trọng cần biết về cách app đang hoạt động:

1. Bảng (table) không dùng Docxtemplater loop — chúng được inject trực tiếp vào XML (table-injector.ts) trước khi Docxtemplater chạy. Docxtemplater chỉ render {{field}} trên văn bản sau khi bảng đã được nhúng vào.

2. Render hiện tại là một pass duy nhất: doc.render(Record<string,string>). Không có array, không có loop. Toàn bộ kiến trúc được thiết kế quanh flat key-value.

3. Batch processing đã bị block cứng cho row_group fields — vì table data không thể flatten thành Excel row. Chorus block phải xử lý tương tự.

1 — Name Rule
Hai namespace tách biệt hoàn toàn bằng hai cặp delimiter khác nhau:

Loại	Cú pháp	Quy tắc tên	Ghi chú
Field thường	{{fieldName}}	camelCase	Giữ nguyên, không thay đổi
Mở chorus block	{%#BLOCK_NAME%}	UPPER_SNAKE_CASE	Gõ trực tiếp trong Word
Đóng chorus block	{%/BLOCK_NAME%}	Tên phải khớp mở	
Biến trong chorus	{%varName%}	camelCase	Chỉ hợp lệ trong block
Lý do chọn {% %} thay vì {{# }} hay angle-bracket:

{{# }} → regex /\{\{([^{}]+)\}\}/g đang dùng sẽ capture #BLOCK_NAME như một field tên — sai ngay tại bước extract
<< >> → đã dùng cho table markers, dễ gây nhầm lẫn cho người tạo template
{% %} → % không phải XML entity, không cần encode; cặp delimiter chưa dùng ở đâu trong codebase; regex /{%([^%]+)%}/g không bao giờ overlap với /{{\{...}\}}/g
Ví dụ template hợp lệ:

Tên hợp đồng: {{tenHopDong}}
Ngày ký: {{ngayKy}}
<<DANH_SACH_HANG_HOA:stt,tenHang,soLuong,donGia>>     ← table (giữ nguyên)
{%#DIEU_KHOAN_THANH_TOAN%}
  Đợt {%dot%}: Bên A thanh toán {%soTien%} vào ngày {%ngayThanhToan%}
  cho hạng mục {%hangMuc%}.
{%/DIEU_KHOAN_THANH_TOAN%}
Người ký: {{nguoiKy}}
2 — Render Mechanism: Two-pass pipeline
Docxtemplater được gọi hai lần tuần tự, mỗi lần với một delimiter riêng:

template.docx
      │
      ▼
[injectTablesIntoZip]          ← giữ nguyên, chạy trước
      │
      ▼
┌─────────────────────────────────────────┐
│  PASS 1 — delimiter: {{ }}              │
│  Input: Record<string, string>          │
│  { tenHopDong, ngayKy, nguoiKy, ... }   │
│  → Thay tất cả {{...}}                  │
│  → {%...%} bị bỏ qua (delimiter khác)  │
└──────────────┬──────────────────────────┘
               │ intermediateBuffer (valid .docx)
               ▼
┌─────────────────────────────────────────┐
│  PASS 2 — delimiter: {% %}              │
│  Input: Record<string, Array<object>>   │
│  { DIEU_KHOAN_THANH_TOAN: [            │
│      { dot, soTien, ngayThanhToan, ... }│
│      { dot, soTien, ngayThanhToan, ... }│
│  ]}                                     │
│  → Thay tất cả {%...%} + loop các      │
│    instance                             │
│  → {{...}} đã biến mất từ Pass 1       │
└──────────────┬──────────────────────────┘
               │ finalBuffer
               ▼
          output.docx
Implementation trong docx-generation.ts:

export function generateDocumentTwoPasses(
  templateBuffer: Buffer,
  fieldValues: Record<string, string>,
  tableDataMap: Record<string, TableData>,
  blockDataMap: Record<string, Array<Record<string, string>>>
): Buffer {
  // Pre-step: inject tables (existing, unchanged)
  let zip = new PizZip(templateBuffer);
  if (Object.keys(tableDataMap).length > 0) {
    zip = injectTablesIntoZip(zip, tableDataMap);
  }
  // Pass 1: top-level fields
  const doc1 = new Docxtemplater(zip, {
    delimiters: { start: '{{', end: '}}' },
    parser: createCustomParser(),       // existing — handles default/checklist
    paragraphLoop: true,
    linebreaks: true,
    nullGetter: () => '',
  });
  doc1.render(processFieldValues(fieldValues));
  const intermediate = doc1.getZip().generate({
    type: 'nodebuffer', compression: 'DEFLATE'
  });
  // Pass 2: chorus blocks (only runs if there are any blocks)
  if (Object.keys(blockDataMap).length === 0) return intermediate;
  const zip2 = new PizZip(intermediate);
  const doc2 = new Docxtemplater(zip2, {
    delimiters: { start: '{%', end: '%}' },
    paragraphLoop: true,
    linebreaks: true,
    nullGetter: () => '',
  });
  doc2.render(blockDataMap);
  return doc2.getZip().generate({ type: 'nodebuffer', compression: 'DEFLATE' });
}
Nếu template không có block nào → blockDataMap rỗng → Pass 2 bị skip hoàn toàn → zero overhead cho documents hiện tại.

3 — Data Flow: Storage ↔ Generation ↔ Export
3a. Storage (Schema thay đổi tối thiểu)
template_tables  (thêm 1 column)
  blockType: 'table' | 'block'   DEFAULT 'table'
  ↑ 1 migration, npm run db:push, existing records unaffected
document_table_data  (không thay đổi gì)
  tableName → matches template_tables.name (kể cả block)
  rows: jsonb  → [{dot, soTien, ngayThanhToan}, ...]
Tái dùng document_table_data 100% — không cần bảng mới, không cần migration phức tạp.

3b. Load data cho Generation
// Trong documents.controller.ts — loadTableDataMap được tách thành 2:
async function loadTableDataMap(documentUuid, templateUuid) {
  const allRows = await db.query(`
    SELECT dtd.table_name, dtd.rows, tt.block_type
    FROM document_table_data dtd
    JOIN template_tables tt
      ON tt.template_uuid = $1 AND tt.name = dtd.table_name
    WHERE dtd.document_uuid = $2
  `, [templateUuid, documentUuid]);
  const tableDataMap: Record<string, TableData> = {};
  const blockDataMap: Record<string, Array<Record<string,string>>> = {};
  for (const row of allRows.rows) {
    if (row.block_type === 'block') {
      blockDataMap[row.table_name] = row.rows;  // đưa vào Pass 2
    } else {
      tableDataMap[row.table_name] = { columns: ..., rows: row.rows };  // đưa vào injectTablesIntoZip
    }
  }
  return { tableDataMap, blockDataMap };
}

5 — An toàn & Edge Cases
Rủi ro	Biện pháp
Block mở không có đóng	Validate tại upload: mọi {%#NAME%} phải có {%/NAME%} cùng tên — trả lỗi chi tiết
Biến trùng tên giữa field và chorus	Warn (không block) lúc upload, ghi rõ tên biến bị trùng
Block lồng nhau	Không hỗ trợ — validate và trả lỗi rõ nếu phát hiện nested blocks
Thay thế file template	Extend replaceTemplateFile validation: field parity + block parity (cùng block names, cùng biến trong block)
Batch processing	Block cứng nếu template có chorus blocks (như đã làm với row_group)
Pass 2 crash	Wrap trong try-catch riêng, error message rõ "Lỗi render chorus BLOCK_NAME"
Cache invalidation	document_table_data update → trigger cache invalidation (hiện tại updateDocument đã invalidate cache — cần đảm bảo update chorus data cũng trigger)
Template preview	Hiển thị một instance rỗng cho mỗi chorus block, wrap bằng đường viền khác biệt
Interactive preview	Chorus blocks hiển thị dạng "read-only" trong interactive mode — scope quá phức tạp cho inline editing
Backward compatibility	Tất cả documents hiện tại không có {%...%} → Pass 2 không chạy → zero impact
